一些基本概念

JMM 的关键技术点是围绕多线程的原子性、可见性和有序性来建立的。因此，首先需要了解这些概念。

原子性

原子性是指一个操作是不可中断的。即使是在多个线程一起执行的时候，一个操作一旦开始，就不会被其它线程干扰。
但是对于 32 位系统来说，long 型(64 位)数据的读写不是原子性的，如果两个线程同时对 long 进行写入的话，对线程之间的结果是有干扰的。

可见性

可见性是指当一个线程修改了某一个共享变量的值，其它线程是否能够立即知道这个修改。

对于串行程序来说，可见性问题是不存在的，因为在任何一个操作步骤中修改了某个变量，那么在后续的步骤中，读取到的变量的值，一定是修改后的新值。

而在并行程序中，如果一个线程修改了某一个全局变量，那么其它线程未必可以马上知道这个改动。如果在 CPU1 和 CPU2 上各运行了一个线程，它们共享变量 t，由于编译器优化或者硬件优化的缘故，在 CPU1 上的线程将变量 t 进行了优化，将其缓存在 cache 或者寄存器里。这种情况下，如果在 CPU2 上的某个线程修改了变量 t 的实际值，那么 CPU1 上的线程可能无法意识到这个改动，因此就产生了可见性问题。

有序性

有序性问题是因为程序在执行时，可能会进行指令重排，重排后的指令与原指令在顺序上未必一致。

假设线程 A 首先执行 write() 方法，接着线程 B 执行 read() 方法，如果发生指令重排，那么线程 B 执行到 int i = a + 1 时，不一定能看到 a 已经被赋值为 1。

指令重排可以保证串行语义的一致性，但没有义务保证多线程间的语义也一致。

那么，为什么要进行指令重排，一步步按顺序执行不好吗？

其实是为了性能考虑。我们知道，一条指令的执行简单地说，可以分为以下 5 步：

  • 取值 IF
  • 译码 ID
  • 执行 EX
  • 存储器访问 MEM
  • 写回 WB

每一个步骤都可能使用不同的硬件完成，可以使用流水线技术来执行指令。如果后一条指令的某个步骤需要前面指令的结果，那么后面执行需要等待（产生气泡），从而性能会受影响。指令重排是把不相关或者不产生影响的指令调到前面，这样，气泡数减少，性能提升。

Happen-Before 规则

Happen-Before 规则是说明哪些指令不能重排。

虽然虚拟机和执行系统会对指令进行一定的重排，但是指令重排是有原则的，并非所有指令都能够随便改变执行的为止，以下是一些基本原则：

  • 程序顺序原则：一个线程内保证语义的串行性
  • volatile 规则：volatile 写先于读，保证 volatile 变量的可见性
  • 锁规则：解锁必然发生在随后的加锁前
  • 传递性：A 先于 B，B 先于 C，那么 A 必然先于 C
  • 线程的 start() 方法先于它的每一个动作
  • 线程的所有操作先于线程的终结
  • 线程的中断先于被中断线程的代码
  • 对象的构造函数执行、结束先于 finalize() 方法

volatile 与 Java 内存模型

volatile 英文解释是“易变的，不稳定的”。当用 volatile 去申明一个变量时，就等于告诉虚拟机，这个变量极有可能被某些程序或者线程修改。为了确保可见性，虚拟机必须采用一些特殊的手段。

volatile 可以保证 long 型数据操作的原子性，但 volatile 不能代替锁，它也无法保证一些复合操作的原子性，如 i++。

线程安全与 synchronized

并行程序开发的一大关注点是线程安全。一般来说，程序并行化是为了获得更高的执行效率，但前提是不能以牺牲正确性为代价。如果程序并行化后，连基本的执行结果的正确性都无法保证，那么并行程序本身也就没有任何意义。

上一小节中，“上述代码的输出总是小于 100000”，这就是线程不安全的栗子。要解决此问题，就必须保证多个线程在对 i 进行操作时完全同步。关键字 synchronized 的作用就是实现线程间的同步。它的工作是对同步代码加锁，使得每一次只能有一个线程进入同步块，从而保证线程间的安全性。

关键字 synchronized 可以有多种用法，

  • 指定加锁对象
  • 直接作用于实例方法：相当于对当前实例加锁
  • 直接作用于静态方法：相当于给当前类加锁

除了用于线程同步、确保线程安全外，synchronized还可以保证线程间的可见性和有序性。

从可见性角度，synchronized 可以完全替代 volatile 的功能；而对于有序性，由于 synchronized 限制每次只有一个线程可以访问同步块，因此，无论同步块内的代码如何被乱序执行(synchronized 无法防止块内指令重排)，只要保证串行语义一致，那么执行结果总是一样的。
